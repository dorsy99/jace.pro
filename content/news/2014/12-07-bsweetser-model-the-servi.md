height: 8pt; padding: 0px;">  </p><p><span style="font-family: Arial;">In the first post in this series, I indicated that Models support the Service Catalog and provide additional value when you begin the asset lifecycle with request through the Service Catalog.</span></p><p><a _jive_internal="true" href="/servlet/JiveServlet/showImage/38-3707-16233/Screen Shot 2014-10-15 at 11.24.46 AM.png"><img  alt="Screen Shot 2014-10-15 at 11.24.46 AM.png" class="image-0 jive-image" height="712" src="85fa0806db9cdb048c8ef4621f96190a.iix" style="height: 384px; width: 620px; display: block; margin-left: auto; margin-right: auto;" width="1150"/></a><span style="font-family: Arial;">With good Model information, you can generate catalog items for end users to request hardware and software in your environment. Some information in the catalog item is maintained in the Model record, such as the Short description, Description, and Picture. Simply click the Related Link to publish to the catalog. This creates the entry end users see in the catalog.</span></p><p style="min-height: 8pt; height: 8pt; padding: 0px;">  </p><p><span style="font-family: Arial;">Because the request is tied to a Product Model, you can identify if you have any of the requested item in stock before you go through the process to procure more. </span></p><p style="min-height: 8pt; height: 8pt; padding: 0px;">  </p><p><span style="font-family: Arial;">When you do need to procure, you can create Vendor Catalog Items to capture the details about the items from your vendors, such as the vendor's part numbers and prices. Then you can select which vendor to purchase from when you source the request.</span></p><p><a _jive_internal="true" href="/servlet/JiveServlet/showImage/38-3707-16234/Models and catalogs.png"><img  alt="Models and catalogs.png" class="image-1 jive-image" height="497" src="963a64cadb909344e9737a9e0f96193d.iix" style="height: 391px; width: 620px; display: block; margin-left: auto; margin-right: auto;" width="789"/></a><span style="font-family: Arial;">As described in a previous post, the Model record represents the <span style="text-decoration: underline;">default configuration</span> for a Model number. With the catalog record defined, you can then identify any options you want to make available as <a title="k-external-small" class="jive-link-external-small" href="http://wiki.servicenow.com/index.php?title=Using_Service_Catalog_Variables" rel="nofollow" target="_blank">Variables</a>. If the options are similar for different Models, you might consider creating a <a title="k-external-small" class="jive-link-external-small" href="http://wiki.servicenow.com/index.php?title=Service_Catalog_Variable_Sets" rel="nofollow" target="_blank">Variable Set</a> to reuse. This allows end users to select options available for the model when they request it.</span></p><p><span style="font-family: Arial;"><br/></span></p><p><a _jive_internal="true" href="/servlet/JiveServlet/showImage/38-3707-16235/Catalog item with options.png"><img  alt="Catalog item with options.png" class="jive-image image-2" height="679" src="d519c806db1017041dcaf3231f9619e5.iix" style="height: 327px; width: 620px; display: block; margin-left: auto; margin-right: auto; border: #000 solid 1px;" width="1287"/></a><span style="font-family: Arial;">Bonus section for usability: Some manufacturers may have a different Model number for every type of configuration, though. Apple is an example of this, where every different iPhone or iPad configuration (color, size) has its own Model number. Rather than listing each of these for your end users to select from in the catalog, you could create an <strong>Order Guide</strong> in ServiceNow for the type of device to presents the various options. With the options selected on the first page of the Order Guide, it then selects the appropriate model of the device.</span></p><p><span style="font-family: Arial;"><br/></span></p><p><a _jive_internal="true" href="/servlet/JiveServlet/showImage/38-3707-16236/Order guide.png"><img  alt="Order guide.png" class="jive-image image-3" height="877" src="26a8ab35dbd41fc068c1fb651f96193d.iix" style="height: 582px; width: 620px; display: block; margin-left: auto; margin-right: auto; border: #000 solid 1px;" width="934"/></a><span style="font-family: Arial;">This post is not meant to be a primer in the Service Catalog, but to provide some thoughts on the benefits on how Models work with it. If you have questions about setting any of these up, please post in the comments.</span></p><p style="min-height: 8pt; height: 8pt; padding: 0px;">  </p><p><span style="font-family: Arial;">In the next couple posts in the series, I will discuss fields in the Hardware Model record, their purpose, and how you can better leverage them in your processes. The first covers default fields and the second covers some new fields I recommend and how you can utilize them.</span></p>
