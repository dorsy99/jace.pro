<p>A simple setting located on sys_properties can cause public pages links to start asking for authentication. The glide.ui.cms.enforce_public_pages setting can be glossy in security but without prior manual steps you may be affecting the convenience of safe public places.</p><p></p><p><img  alt="weblockdown1.png" class="image-2 jive-image" height="128" src="f26995c6db18d304b322f4621f961970.iix" style="width: 128px; height: 128px; display: block; margin-left: auto; margin-right: auto;" width="128"/></p><p></p><p></p><p>I will cover:</p><ul><li><a title="" _jive_internal="true" href="/community?id=community_blog&sys_id=a90eaa2ddbd0dbc01dcaf3231f96191f">Content Management System (CMS) and Single Sign-on (SSO)</a></li><li>How system property glide.ui.cms.enforce_public_pages affects public pages</li><li>Public pages are asking for authentication</li></ul><p></p><h2>Content Management System (CMS) and Single Sign-on (SSO)</h2><p>A site is a group of related content pages with a consistent look and feel, defined by a common layout, theme, and URL suffix.</p><p>Before creating a site for the first time, review the <a class="mw-redirect" href="http://wiki.servicenow.com/index.php?title=Content_Management_Overview" title="Content Management Overview">Content Management System</a> page to understand the concepts and principles of using CMS.</p><p></p><p>SSO authentication mainly provided by our newly added <a title="ki.servicenow.com/index.php?title=Multiple_Provider_Single_Sign-On" href="http://wiki.servicenow.com/index.php?title=Multiple_Provider_Single_Sign-On">Multiple Provider Single Sign-On</a> plugin allows organizations to use several SSO identity providers (IdPs) to <a title="ki.servicenow.com/index.php?title=Authentication" href="http://wiki.servicenow.com/index.php?title=Authentication">manage authentication</a> as well as retain local database (basic) authentication. The integration supports any combination of local and external authentication methods on a single instance:</p><ul><li>SAML 2.0</li><li>Digest Authentication</li><li>LDAP</li><li>Local database authentication</li></ul><p>For example, a globally dispersed corporation might require one SSO provider for their employees, a different one for their vendors, and local database authentication for their administrators. Administrators can configure Multi-Provider SSO for <a href="http://wiki.servicenow.com/index.php?title=Creating_Users_and_Associating_to_a_Group#Creating_a_User" title="Creating Users and Associating to a Group">individual users</a> or for all users who belong to a <a href="http://wiki.servicenow.com/index.php?title=Adding_a_New_Company" title="Adding a New Company">company</a>. You cannot configure Multi-Provider SSO for <a href="http://wiki.servicenow.com/index.php?title=Creating_Groups" title="Creating Groups">groups</a>.</p><p></p><p>Since public pages do not require login, SSO is skipped and CMS sites are public by default.This is because the pages has been defined with<strong> Active </strong>set to<strong> True</strong> on the <strong>sys_public table.</strong></p><p></p><p><img   alt="2016-11-30_1746.png" class="image-1 jive-image" src="9286b73ddb50d3041dcaf3231f9619df.iix" style="width: 620px; height: 165px;"/></p><p></p><p>You can configure your CMS pages to use SSO by making the view_content page private (active = false). Making a CMS page is a common request as companies and organizations want to make it available for internal users or specific people.</p><p></p><pre __default_attr="info" __jive_macro_name="alert" alert="info" class="jive_text_macro jive_macro_alert" data-renderedposition="1090.199951171875_8_917_64"><p>To turn off public-facing content pages and enable SSO: Navigate to <a href="http://wiki.servicenow.com/index.php?title=Making_a_Page_Public" title="Making a Page Public">Public Pages</a> (sys_public.list) the for the "view_content" page, set <strong>Active</strong> to <strong>false</strong>.</p></pre><p></p><p>However, there are some pages that need to remain public. e.g. A page that contains the link to the request for password resets. There is a system property that cause the public CMS pages defined as public to enforce the validation of its internal links as public.</p><p></p><h2>How system property glide.ui.cms.enforce_public_pages affecting public pages</h2><p>This property on sys_property table can be created and set as   <span style="color: #3334ca;">"glide.ui.cms.enforce_public_pages" = true</span> (recommended). The value "true" enforces the links defined inside CMS public pages need to be defined on <a title="ki.servicenow.com/index.php?title=Making_a_Page_Public#gsc.tab=0" href="http://wiki.servicenow.com/index.php?title=Making_a_Page_Public#gsc.tab=0">sys_public</a> table as well.</p><p></p><p>To validate the pages access search on <strong>Content management &gt; Pages</strong> for pages with 'Content site' is empty. Those are the pages to open and validate the '<a title="ki.servicenow.com/index.php?title=Content_Management_Security#gsc.tab=0" href="http://wiki.servicenow.com/index.php?title=Content_Management_Security#gsc.tab=0">read roles</a>'. Then view each page with the "view page" link and take note of all the pages that need to be public or not. The public links need to be added to the <a title="ki.servicenow.com/index.php?title=Making_a_Page_Public#gsc.tab=0" href="http://wiki.servicenow.com/index.php?title=Making_a_Page_Public#gsc.tab=0">sys_public</a> table. Ensure the pages that are configured for public viewing do not include any sensitive information.</p><p></p><p style="text-align: center;"><em>See <a href="https://hi.service-now.com/kb_view.do?sysparm_article=KB0547450" title="https://hi.service-now.com/kb_view.do?sysparm_article=KB0547450">ServiceNow KB: Configuring and Enforcing public CMS pages (KB0547450)   </a>for more detail (Requires a HI login to view)</em></p><p></p><p>If the property is not defined or false, the links associated within the public CMS page are consider public. By setting it to "false," it puts you at risk for information disclosure vulnerability.</p><p></p><p></p><h2>Public pages are asking for authentication</h2><p>Prior to introducing this property, public CMS pages will not ask for authentication. With the new property <span style="color: #3334ca;">"glide.ui.cms.enforce_public_pages" = true</span> (recommended) in place, if the links within the public CMS pages are not explicitly "public", our system will issue a request for <a title="ki.servicenow.com/index.php?title=Authentication#gsc.tab=0" href="http://wiki.servicenow.com/index.php?title=Authentication#gsc.tab=0">authentication</a>. To make those pages public, you need to set or add a record for the page into the sys_public table with Active set to true.</p><p></p><p>If you do go the route of setting pages to public, double-check that sensitive pages are left private and explicitly set public pages on sys_public.</p><p></p><p>For more information on CMS and public pages prompting a login see:</p><ul><li><a
