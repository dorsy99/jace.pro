<p>When we need additional information on a known problem that is affecting customers or our own instances, we make good use of our <img   alt="Internet_Explorer_9_icon.svg.png" class="image-0 jive-image" height="134" src="38580c46db1cdb048c8ef4621f9619fe.iix" style="font-size: 13.3333330154419px; height: 134px; width: 134px; float: right;" width="134"/><span style="font-size: 10pt; line-height: 1.5em;">extensive Known </span><span style="font-size: 10pt; line-height: 1.5em;">Error </span><span style="font-size: 10pt; line-height: 1.5em;">Knowledge Base, which is updated daily. These articles help identify issues that we are aware of and are either investigating, fixing, or offering a workaround </span><span style="font-size: 10pt; line-height: 1.5em;">solution. Internet Explorer navigation of the product tends to be a hot topic, so we try to address </span><span style="font-size: 10pt; line-height: 1.5em;">these as quickly as possible. Most of the known errors and issues in this list are around the user interface acting up when using the latest version of IE.</span></p><div><p></p><p></p><h3>Here are the Top KE articles on Internet Explorer issues that ServiceNow Employees referenced in July:</h3><p></p><h4>1) <a title="i.service-now.com/kb_view.do?sysparm_article=KB0535211" href="https://hi.service-now.com/kb_view.do?sysparm_article=KB0535211">Internet Explorer 11 clears content of second password field on a form</a></h4><p><span style="font-size: 10pt; line-height: 1.5em;">ServiceNow has recently discovered a bug in Internet Explorer 11 that, under very specific circumstances, causes password input fields to be cleared. Using IE 11 to visit any form that has two password input fields will cause the second password input field to be cleared when saving the form, resulting in loss of data in that field. As a workaround, use any other browser, including older versions of Internet Explorer. If you are required to use IE 11, a workaround is being discussed on Microsoft's forums </span><a title="cial.msdn.microsoft.com/Forums/ie/en-US/7d02173f-8f45-4a74-90bf-5dfbd8f9c1de/ie-11-issue-with-two-password-input-fields?forum=iewebdevelopment" href="http://social.msdn.microsoft.com/Forums/ie/en-US/7d02173f-8f45-4a74-90bf-5dfbd8f9c1de/ie-11-issue-with-two-password-input-fields?forum=iewebdevelopment" style="font-size: 10pt; line-height: 1.5em;">here</a><span style="font-size: 10pt; line-height: 1.5em;">. For your convenience, we have provided instructions in this Known Error.</span></p><p></p><h4>2) <a title="i.service-now.com/kb_view.do?sysparm_article=KB0538372" href="https://hi.service-now.com/kb_view.do?sysparm_article=KB0538372">When using Internet Explorer while logged into HI, the browser is forced into compatibility mode</a></h4><p><span style="font-size: 10pt; line-height: 1.5em;">Under certain conditions, customers who use Internet Explorer to log in to HI.service-now.com have experienced their browsers running in compatibility mode. This condition is caused by *.service-now.com or HI.service-now.com being added to the </span><span style="font-size: 10pt; line-height: 1.5em;"><strong>Trusted sites</strong></span><span style="font-size: 10pt; line-height: 1.5em;"> list within Internet Explorer. To resolve this issue, remove the references to ServiceNow in the </span><span style="font-size: 10pt; line-height: 1.5em;"><strong>Trusted sites</strong></span><span style="font-size: 10pt; line-height: 1.5em;"> zone.</span></p><p></p><h4>3) <a title="i.service-now.com/kb_view.do?sysparm_article=KB0541718" href="https://hi.service-now.com/kb_view.do?sysparm_article=KB0541718">When using Internet Explorer, the UI14 interface does not work in Eureka when document mode/user agent string is less than 9 or compatibility view is enabled</a></h4><p><span style="font-size: 10pt; line-height: 1.5em;">Document mode in F12 Developer Tools is set to 7 by default on IE 9, 10, and 11 if service-now.com is treated as an intranet site. This causes UI14 to fail when accessing a Eureka instance. Users should try to figure out why service-now.com is being treated as an intranet site on their local machine. To resolve this issue, change the settings in Document mode and User agent string or clear the Display intranet sites in Compatibility View option. See the Workaround section of this Known Error for detailed instructions.</span></p><p></p><h4>4) <a title="i.service-now.com/kb_view.do?sysparm_article=KB0533467" href="https://hi.service-now.com/kb_view.do?sysparm_article=KB0533467">Edit UI Action menu in the top form button is hidden behind the form header (in IE)</a></h4><p><span style="font-size: 10pt; line-height: 1.5em;">When trying to edit UI actions on a form in Internet Explorer, the </span><span style="font-size: 10pt; line-height: 1.5em;"><strong>Edit UI Action. </strong></span><span style="font-size: 10pt; line-height: 1.5em;"> context menu is hidden behind the form header and only a very small portion of the button is displayed. In Google Chrome, right-clicking a form button on the top of the form does not render the </span><span style="font-size: 10pt; line-height: 1.5em;"><strong>Edit UI Action</strong></span><span style="font-size: 10pt; line-height: 1.5em;"> context menu. To workaround this issue in IE, users can right-click the buttons to show the full menu, or choose actions from the Personalize &gt; UI Actions option.</span></p><p></p><h4>4) <a title="i.service-now.com/kb_view.do?sysparm_article=KB0535116" href="https://hi.service-now.com/kb_view.do?sysparm_article=KB0535116">In IE8 and IE9, click-through is not working on the Approvers form for a problem</a></h4><p><span style="font-size: 10pt; line-height: 1.5em;">Clicking through reference icons in the Problem &gt; Approval form in IE8 and IE9 does not navigate to reference documents as expected. There is no workaround for this issue, so we recommend using Chrome.</span></p><p><span style="font-size: 10pt; line-height: 1.5em;"><br/></span></p><p><span style="font-size: 10pt; line-height: 1.5em;"><br/></span></p><p><span style="font-size: 10pt; line-height: 1.5em;"><br/></span></p></div>
