<p>Are you already using <a title="ki.servicenow.com/index.php?title=UI_Actions" href="http://wiki.servicenow.com/index.php?title=UI_Actions">UI actions</a>? Many ServiceNow customers use them in a variety of creative ways. If you are using UI actions in your instance, consider refining them and controlling when they appear and for whom. For example, you can show the <strong>Delete</strong> button on the <strong>Problem</strong> form only to users with the admin role instead of all users.</p><p></p><p>For those who are not using them yet, a brief definition. UI actions are elements such as buttons, links, or a context menu that you can add to a form or list. When a user clicks the element, some script is executed. UI actions help customize the user interface and make it more specific for individual users. Users with the admin or ui_action_admin role can create UI actions.</p><p></p><p><img   alt="ui_actions1.jpg" class="image-0 jive-image" src="ae4f10cadb9c9fc068c1fb651f96199a.iix" style="height: auto; display: block; margin-left: auto; margin-right: auto;"/></p><p></p><p></p><h2>How to limit access with conditions:</h2><p><span style="color: #333333; font-family: Omnes-pro, Arial, Verdana, sans-serif; font-size: 14px;">Many users limit a UI action by defining a condition. If the condition is <strong>true</strong>, the action is visible. (Note that leaving the field empty defaults the condition to true.) </span></p><p><span style="color: #333333; font-family: Omnes-pro, Arial, Verdana, sans-serif; font-size: 14px;"><br/></span></p><p><span style="color: #333333; font-family: Omnes-pro, Arial, Verdana, sans-serif; font-size: 14px;">Here is an example for the Delete button on a Problem record:</span></p><p><span style="color: #333333; font-family: Omnes-pro, Arial, Verdana, sans-serif; font-size: 14px;"><img   alt="ui_actions2.jpg" class="image-1 jive-image" src="e5dfbffddb18d3049c9ffb651f9619d2.iix" style="height: 367px; width: 620px; display: block; margin-left: auto; margin-right: auto;"/><br/></span></p><p>In this UI action, the Delete button only appears on the Problem record if the problem is in Closed/Resolved state.</p><p></p><p></p><h2>How to limit access with roles:</h2><p><span style="font-size: 10pt; line-height: 1.5em;">Combine conditions and roles to limit access to UI actions by user role. The syntax is: </span><span style="font-family: 'courier new', courier;">gs.hasRole('role_name')</span><span style="font-size: 10pt; line-height: 1.5em;">.</span><span style="font-size: 10pt; line-height: 1.5em;">Here is an example again for the Delete button on the Problem record, but this time we are controlling access by role:</span></p><p><img   alt="ui_actions3.jpg" class="jive-image image-2" src="256cf042db9497041dcaf3231f9619e3.iix" style="height: 349px; width: 620px; display: block; margin-left: auto; margin-right: auto;"/></p><p><span style="font-size: 10pt; line-height: 1.5em;">In this UI action, the Delete button only appears on the Problem record if the user has the admin or problem_admin role.</span></p><p></p><h3></h3><h2>How to limit access with views:</h2><p>Another nifty way to control who sees a UI action is by including or excluding views using the UI Action Visibility related list. Note that if you do not include or exclude views, the action appears on all views. In this example, we are specifying that the Close Incident UI Action is only included (visible) in Advanced view:</p><p><img   alt="ui_actions4.jpg" class="jive-image image-3"
