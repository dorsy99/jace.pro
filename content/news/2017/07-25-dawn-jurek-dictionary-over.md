<p>ServiceNow provides you with the flexibility to customize forms so that they can mirror your business processes. But sometimes tinkering with default values can be tricky. In this installment of our <a title="" _jive_internal="true" href="/content?filterID=all~objecttype~objecttype%5Bblogpost%5D&amp;filterID=all~tag%5Bnowsupport+best+practices+series%5D&amp;query=nowsupport+best+practices" style="font-family: arial, sans-serif; color: #3778c7;">best practices series</a>, we look at how changing the default <strong>State</strong> field on one form can affect other forms as well. But not to worry: we also show you how to use the magic of dictionary overrides to avoid this pitfall.</p><p></p><p>Consider this <em><strong>Oops</strong></em> scenario:</p><p></p><p>In the base system, the default value for the <strong>State</strong> field on the Incident form is <strong>New</strong>, but you want the default value to be <strong>In Progress</strong> instead:</p><p><img   alt="Incident form showing State choices final.jpg" class="image-1 jive-image" src="2b882d86dbdc130468c1fb651f9619d2.iix" style="width: 800px; height: 214px; display: block; margin-left: auto; margin-right: auto;"/></p><p></p><p>So you right-click the <strong>State</strong> label to show the choice list for this field, and filter it for the Incident table. You note that the <strong>Value</strong> corresponding to <strong>In Progress</strong> is <strong>2</strong>:</p><p style="text-align: center;"><img   alt="Choices for State field on Incident table final.jpg" class="image-2 jive-image" src="fcb1c1c6dbd09f048c8ef4621f961968.iix" style="width: 800px; height: 350px;"/></p><p></p><p>Then you go back and right-click the <strong>State</strong> field on the Incident form again to configure the dictionary for this field, changing its <strong>Default value</strong> to <strong>2</strong>:</p><p style="text-align: center;"><img   alt="Change default value to 2 final.jpg" class="image-3 jive-image" src="0aae1042db5c1f048c8ef4621f9619d0.iix" style="width: 800px; height: 451px;"/></p><p></p><p>It works! The default <strong>State</strong> value for new incidents is now <strong>In Progress</strong>. But there are also some unintended consequences. For example, the Problem form now displays a default <strong>State</strong> of <strong>Known Error</strong>. That's not right!</p><p style="text-align: center;"><img   alt="Problem form showing default State as Known Error final.jpg" class="image-4 jive-image" src="c42e3fb1db585704ed6af3231f96198e.iix" style="width: 800px; height: 148px;"/></p><p></p><p>And some other forms are affected as well.</p><p></p><h2>What went wrong?</h2><p></p><p>If you had looked carefully at the dictionary entry for the <strong>State</strong> field when you changed the default value, you would have noticed that the change you made affected the Task table—not just the Incident table:</p><p style="text-align: center;"><img   alt="Dictionary entry for State field-Task table final.jpg" class="image-5 jive-image" src="58b1a54adb50df048c8ef4621f9619e8.iix" style="width: 800px; height: 256px;"/></p><p></p><p>That's because the Incident table is a <a title="ocs.servicenow.com/bundle/istanbul-servicenow-platform/page/administer/reference-pages/reference/r_TablesAndClasses.html" href="https://docs.servicenow.com/bundle/istanbul-servicenow-platform/page/administer/reference-pages/reference/r_TablesAndClasses.html" style="color: #954f72;">child class</a> whose parent is the Task table. The Incident table inherited the <strong>State</strong> field from the Task table, and any changes made to this dictionary entry affect the Task table and all <a title="outu.be/wSqg05k65Ew?t=4m19s" href="https://youtu.be/wSqg05k65Ew?t=4m19s">child tables</a> extending from it. So, changing the default value of the <strong>State</strong> field on the Task table also changed the default <strong>State</strong> value for a bunch of other tables, including the Problem table.</p><p></p><p>The proper way to change the default <strong>State</strong> value for the Incident table without affecting the Task table or its extended tables is to use a <em>dictionary override</em>. First, we change that default <strong>State</strong> value back to <strong>1</strong>, which is what it should be for most forms.</p><p></p><h2>What are dictionary overrides and how do I create them?</h2><p></p><p><a title="ocs.servicenow.com/bundle/jakarta-servicenow-platform/page/administer/data-dictionary-tables/concept/c_DictionaryOverrides.html" href="https://docs.servicenow.com/bundle/jakarta-servicenow-platform/page/administer/data-dictionary-tables/concept/c_DictionaryOverrides.html">Dictionary overrides</a> allow system administrators to define certain field settings on an extended table differently from those on the parent table without affecting the parent table or its extended tables. Dictionary overrides are defined in the dictionary entry record for the field on the parent table. For example, here's the dictionary entry for the <strong>State</strong> field on the Task table, scrolled down to display the Dictionary Overrides related list:</p><p style="text-align: center;"><img   alt="Dictionary Overrides for State field on the Task table final.jpg" class="image-6 jive-image" src="d9834d82db149f048c8ef4621f96192d.iix" style="width: 800px; height: 403px;"/></p><p></p><p>Two dictionary overrides already exist for the <strong>State</strong> field on the Task table—one applies to the Change Request table and the other to the Standard Change Proposal table.</p><p></p><p>To define a new dictionary override to set the default <strong>State</strong> value on the Incident table to <strong>2</strong>, click <strong>New</strong> and fill in the form, like this:</p><p style="text-align: center;"><img   alt="Define dictionary override for Incident Table final.jpg" class="image-7 jive-image" src="d205c082db9c5704ed6af3231f961998.iix" style="width: 800px; height: 491px;"/></p><p></p><p>Here you can see which other settings can be overridden for the <strong>State</strong> field. As check boxes are selected, additional fields are displayed to specify the details for each override.</p><p></p><p>Now, with this new dictionary override, the incident form displays a default <strong>State</strong> value of <strong>2</strong>, and the Problem form displays a default <strong>State</strong> value of <strong>1</strong>, exactly as they should:</p><p style="text-align: center;"><img   alt="Incident-new record with State value of 2 final.jpg" class="image-8 jive-image" src="77e2aff5dbd01fc068c1fb651f961986.iix" style="width: 800px; height: 138px;"/></p><p style="text-align: center;"><img   alt="Problem-new record with State value of 1 final.jpg" class="image-9 jive-image" src="e095454adbd013043eb27a9e0f961956.iix" style="width: 800px; height: 166px;"/></p><p></p><h2>How dictionary overrides affect extended tables</h2><p></p><p>Dictionary overrides are inherited by extended tables, so it may be necessary to define additional overrides to change the value of fields on extended tables. For example, if you define the default value of <strong>cmdb_ci.install_status</strong> to be <strong>3</strong>, and create an override for the same field on <strong>cmdb_ci_hardware</strong> to be <strong>5</strong>, all tables extended from the Hardware table will also default to 5. So if you want the default <strong>cmdb_ci_computer.install_status</strong> to be <strong>3</strong>, you will need another override.</p><p></p><h3><strong>Best Practice: Align state values across applications</strong></h3><p></p><p>A related best practice is to align the <strong>State</strong> field values. These aren't always aligned in the base system, but generally, a <strong>State</strong> of <strong>3</strong> means closed, regardless of whether the record is an incident, a problem, or a change.   If you alter the states, try to keep them aligned across all the applications. This may be easier said than done, but it is useful. For example, you can then make a report on the Task table, and see all closed tasks with one filter, rather than lots of OR conditions.</p><p></p><p>Want to learn more about tables, records and fields?
